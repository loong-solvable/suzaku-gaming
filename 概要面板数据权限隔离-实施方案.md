# 概要面板数据权限隔离 — 深度实施方案（按当前项目代码校准）

> 日期：2026-03-03
> 目标：让 `Dashboard` 严格按当前登录用户权限，且仅基于 **已审核通过（approved）** 的归因绑定申请，返回统计数据。
> 结论：当前项目 **不符合** 该需求；若要做得正确且稳定，改动不应只停留在 `dashboard.service.ts`，还必须处理 `binding_applies` 的一致性风险和查询性能风险。

---

## 0. 执行摘要

本方案基于当前仓库的真实实现得出以下结论：

1. 当前 `Dashboard` 完全按全量 `roles/orders` 统计，没有任何用户级权限过滤。
2. 当前 `Dashboard` 不依赖 `binding_applies`，因此也没有 `approved` 状态过滤。
3. 如果直接把 `Dashboard` 改成"从 `binding_applies(status='approved')` 派生统计"，**还不够稳**，因为当前 `AuditService.deleteBindingApply()` 允许管理员删除任意状态申请，包含 `approved`。
4. 生产可用的方案应采用：
   - **后端单点改造**：控制器透传当前用户、服务层改为"已通过申请作用域统计"
   - **删除语义加固**：禁止删除 `approved` 申请，避免 Dashboard 统计真相源被随意抹掉
   - **索引补强**：当前 `binding_applies` 对新查询路径索引不足，数据量上来后会明显退化

本方案优先保证：

- 统计语义正确
- 权限边界清晰
- 零数据返回稳定
- 对当前前端契约零破坏
- 对后续数据增长具备更好的可扩展性

## 1. 当前代码现状与问题定位

## 1.1 Dashboard 当前实现（与需求直接冲突）

当前后端控制器：

- 文件：`suzaku-gaming-server/src/modules/dashboard/dashboard.controller.ts`
- 现状：`getStatistics()` **没有接收当前用户**
- 实际行为：只要通过 `@Roles` 鉴权，任何角色拿到的都是同一份统计

当前后端服务：

- 文件：`suzaku-gaming-server/src/modules/dashboard/dashboard.service.ts`
- 现状：
  - `getStatistics()` 无入参
  - `getTodayStats()` / `getMonthlyStats()` / `getTotalStats()` 都直接查全量 `prisma.role` / `prisma.order`
  - 不依赖 `bindingApply`
  - 不过滤 `approved`

具体表现：

- `newPlayers` 来自 `roles.registerTime`
- `paidPlayers` / `paidAmount` 来自 `orders.payTime`
- 仅过滤 `isSandbox = false`
- 没有任何 `roleId` 权限作用域

因此，以下需求全部未满足：

- 管理员 / 组长 / 组员视角隔离
- 仅统计 `approved`
- `pending/rejected` 不计入
- "当前权限范围内零数据返回 0"

## 1.2 归因申请当前实现（可作为权限作用域的真相源）

当前归因申请主表：

- 文件：`suzaku-gaming-server/prisma/schema.prisma`
- 表：`BindingApply`（映射 `binding_applies`）

关键字段：

- `roleId`
- `platform`（新语义：组编码 / `cpsGroupCode`）
- `teamLeader`（旧语义兼容：组长用户名）
- `teamMember`
- `applicant`
- `status`（`pending` / `approved` / `rejected`）
- `reviewTime`

这张表天然适合作为"哪些角色已进入归因统计"的真相源，原因是：

1. 它记录了审核结果（`approved`）
2. 它记录了角色归属（`platform` / `teamLeader`）
3. 它记录了申请人（`applicant`）
4. 它是权限判断最接近业务语义的地方

### 业务流程补充说明：同一 roleId 的多次申请

当前 `createBindingApply` 的防重复逻辑（`audit.service.ts`）：

```ts
const existingApply = await this.prisma.bindingApply.findFirst({
  where: { roleId: dto.roleId, status: { in: ['pending', 'approved'] } },
});
```

这意味着：

- 已有 `pending` 或 `approved` 状态的 roleId 不可重复提交
- 但 `rejected` 状态的可以重新提交并最终被 approve
- 因此同一个 roleId 在 `binding_applies` 中可能存在多条记录（如：一条 `rejected` + 一条 `approved`）

本方案使用 `SELECT DISTINCT ba.role_id ... WHERE ba.status = 'approved'` 来构建作用域，已天然处理此场景：无论有多少条历史记录，只要至少一条是 `approved`，该 roleId 就进入统计；`DISTINCT` 确保不会重复计入。

## 1.3 Audit 当前权限隔离逻辑（可复用思想，不能直接照抄）

当前 `AuditService` 已有一套申请列表的数据隔离逻辑：

- 文件：`suzaku-gaming-server/src/modules/audit/audit.service.ts`
- 方法：`buildDataFilter(currentUser)`

现有规则：

- `admin`：无过滤
- `manager`：`platform = cpsGroupCode` 或 `teamLeader = username` 或 `applicant = username`
- `operator`：`applicant = username`

这个逻辑适用于"查看申请列表"，但 **不应直接用于 Dashboard**，原因如下：

1. 你的需求对 `manager` 的定义是"本组（含自身及下属）"，不是"本人提交的一切申请"。
2. 如果保留 `manager` 的 `applicant = username` 条件，理论上会把"本人提交但不属于本组"的历史异常数据也算进去，统计语义被放宽。

结论：

- 可以复用 `platform/teamLeader` 这套新旧兼容思路
- **不能**直接复用 `buildDataFilter()` 作为 Dashboard 过滤器

## 1.4 当前存在的关键一致性风险：approved 申请可被直接删除

当前删除接口：

- 文件：`suzaku-gaming-server/src/modules/audit/audit.controller.ts`
- 路由：`DELETE /audit/binding-applies/:id`
- 权限：`@Roles('admin')`

当前服务实现：

- 文件：`suzaku-gaming-server/src/modules/audit/audit.service.ts`
- 方法：`deleteBindingApply(id)`
- 现状：**不区分状态，找到就删**（且不接收 `currentUser` 参数）

这意味着：

- 一个 `approved` 申请可以被管理员直接删除
- 但 `reviewBindingApply(approve)` 在审批通过时已经把 `roles.cpsGroup/cpsVisible` 写入本地 `Role`
- 删除申请不会回滚 `Role` 的 CPS 字段

如果 Dashboard 按你的需求改为"只认 `approved binding_applies`"，而仍允许删除 `approved`：

- Dashboard 统计会立刻减少
- 但角色本地仍然是"已绑定"
- 申请状态真相源与角色状态真相源发生分裂

这是本次实现必须正面处理的 P0 风险。

## 1.5 当前索引覆盖不足（功能能做，但规模上来会慢）

当前 `binding_applies` 已有索引：

- `status`
- `applicant`
- `applyTime`
- `project`

当前 **缺失** 与新查询路径直接相关的索引：

- `roleId`
- `platform`
- `teamLeader`
- 组合索引（如 `status + platform`）

而新 Dashboard 查询会高度依赖：

- `status = 'approved'`
- `platform = currentUser.cpsGroupCode`
- `teamLeader = currentUser.username`
- `applicant = currentUser.username`
- 去重 `roleId`

所以如果只改代码不补索引，功能可上线，但随着 `binding_applies` 数据增长，性能会明显退化。

## 2. 需求在当前项目中的精确定义（避免语义歧义）

为了避免实现时出现"看似符合、实则口径偏差"的问题，建议将本需求在当前项目中明确为以下语义：

## 2.1 统计对象

Dashboard 统计的不是全量玩家/订单，而是：

> "当前用户权限范围内，所有 `status = 'approved'` 的 `binding_applies.roleId` 所对应的角色和订单数据"

换句话说，先确定 **可计入的 roleId 集合**，再对这些角色/订单做统计。

## 2.2 时间口径

保留现有 Dashboard 的时间维度定义，不在本次需求中改变指标本身：

- `today`：按 `roles.registerTime` / `orders.payTime` 的今日范围统计
- `monthly`：按本月范围统计
- `total`：历史累计

关键说明：

- 本次只改变"**谁能进统计集合**"，不改变"**时间怎么切**"
- 不以 `binding_applies.applyTime` 或 `reviewTime` 作为统计时间轴

这意味着：

- 某角色今天才通过审批，但其本月更早的注册/付费数据，只要属于当前统计窗口，仍会计入 `monthly` / `total`

**时间边界约定**：统一采用左闭右开区间（`>= start AND < end`），避免因 timestamp 精度不一致导致边界数据遗漏或重复统计：

- `today`：`>= 今日 00:00:00` 且 `< 明日 00:00:00`
- `monthly`：`>= 本月 1 日 00:00:00` 且 `< 下月 1 日 00:00:00`

这是最符合现有 Dashboard 结构、改动最小且语义一致的实现。

## 2.3 `activePlayers` 口径（保持不动，仅加权限与 approved 过滤）

当前 `activePlayers` 并不是真实活跃用户口径，而是临时近似。现有代码实际行为：

```ts
// dashboard.service.ts 各时段统计方法中的写法
activePlayers: newPlayers || paidPlayersResult.length
```

即：当 `newPlayers` 为 0（falsy）时 fallback 到 `paidPlayers`。虽然业务上不理想，但本需求没有重定义该指标。

因此本次建议：

- **不改变** `activePlayers` 业务定义
- 仅把它限定在新的"可计入 roleId 集合"内
- TS 层组装时保持等价逻辑：`activePlayers = newPlayers || paidPlayers`

如果要修正为真实活跃口径，应单独立需求，不与本次权限隔离合并。

## 2.4 manager 的权限边界（比申请列表更严格）

对于 Dashboard，`manager` 应仅统计：

- 本组 `platform = currentUser.cpsGroupCode` 的新数据
- 旧数据兼容：`teamLeader = currentUser.username`

**不额外包含** `applicant = currentUser.username`。

原因：

- Dashboard 需求是"本组数据"
- 不是"本人提交的所有数据"

这样可避免把历史异常数据或越权数据误算进来。

## 3. 推荐实现方案（首选：数据库内集合化统计）

## 3.1 总体设计

推荐采用 **单接口、后端服务层重构、数据库内集合化计算** 的方式：

1. Controller 把当前登录用户传入 Service
2. Service 在数据库内先构建"当前用户可见的 approved roleId 集合"
3. 再基于这个集合，直接在数据库内算出 `today/monthly/total` 的统计结果
4. 返回结构保持不变，前端零改动

与简单的"两步法（先查 roleId 列表，再 `IN (...)` 统计）"相比，这种方式更稳健：

- 不把大量 `roleId` 拉到 Node.js 内存
- 避免超长 `IN` 条件
- 减少数据库往返次数
- 更适合后续数据增长

## 3.2 控制器改造（必须）

### 涉及文件

- `suzaku-gaming-server/src/modules/dashboard/dashboard.controller.ts`

### 具体改动

将当前用户透传到服务层。

**推荐方式：创建公共 `@CurrentUser()` 参数装饰器**

当前项目中**不存在** `@CurrentUser()` 装饰器（`src/common/decorators/` 下仅有 `@Public` 和 `@Roles`）。现有控制器通过 `@Req() req` + `(req as any).user` 获取用户，缺乏类型安全。

建议新增公共装饰器和类型：

1. **新建 `src/common/interfaces/current-user.interface.ts`**（提取公共类型，避免各模块重复定义）：

```ts
export interface CurrentUser {
  id: number;
  username: string;
  realName: string;
  role: string;
  level: number;
  parentId: number | null;
  cpsGroupCode: string | null;
  status: number;
}
```

> 注意：此类型须与 `jwt.strategy.ts` 的 `validate()` 返回值对齐。当前 validate 返回 `{ id, username, realName, role, level, parentId, cpsGroupCode, status }`。

2. **新建 `src/common/decorators/current-user.decorator.ts`**：

```ts
import { createParamDecorator, ExecutionContext } from '@nestjs/common';

export const CurrentUser = createParamDecorator(
  (data: unknown, ctx: ExecutionContext) => {
    const request = ctx.switchToHttp().getRequest();
    return request.user;
  },
);
```

3. **改造 Controller**：

```ts
import { CurrentUser } from '../../common/decorators/current-user.decorator';
import { CurrentUser as CurrentUserType } from '../../common/interfaces/current-user.interface';

@Get('statistics')
@Roles('admin', 'manager', 'operator')
async getStatistics(@CurrentUser() user: CurrentUserType) {
  return this.dashboardService.getStatistics(user);
}
```

### 原因

当前 JWT 鉴权后的 `req.user` 已包含完整用户信息（来源：`jwt.strategy.ts` 的 `validate()` 方法从数据库 select 后返回）。使用 `@CurrentUser()` 装饰器比 `@Req()` + 类型断言更安全、更符合 NestJS 惯例，且可被所有需要当前用户的控制器复用。

### 附加建议

在创建公共 `CurrentUser` 类型后，`audit.service.ts` 中已有的同名 `interface CurrentUser`（第 35-41 行）应删除并改为引用公共类型，避免类型分裂。该接口当前缺少 `realName` 和 `status` 字段，与 JWT 实际返回不一致。

## 3.3 服务层改造（必须，核心）

### 涉及文件

- `suzaku-gaming-server/src/modules/dashboard/dashboard.service.ts`

### 目标

将当前：

- `getStatistics()`
- `getTodayStats()`
- `getMonthlyStats()`
- `getTotalStats()`

重构为：

1. `getStatistics(currentUser)`
2. `buildScopedQuery(currentUser, todayStart, tomorrow, monthStart, nextMonth)`
3. `mapMetricsToResponse(row)`

### 推荐实现：使用 `prisma.$queryRaw` + CTE

推荐使用 PostgreSQL CTE 一次性完成作用域构建与统计，不建议采用"Node 层先拿 roleId 数组再统计"的方式作为最终方案。

#### 3.3.1 作用域 CTE

先定义 `scoped_roles`：

```sql
WITH scoped_roles AS (
  SELECT DISTINCT ba.role_id
  FROM binding_applies ba
  WHERE ba.status = 'approved'
    AND ...权限过滤...
)
```

权限过滤按角色生成：

- `admin`

```sql
TRUE
```

- `manager`

```sql
(
  ba.platform = :cpsGroupCode
  OR ba.team_leader = :username
)
```

- `operator`

```sql
ba.applicant = :username
```

#### 3.3.2 统计 SQL（推荐一次返回全部 9 个原始指标）

在同一个 SQL 中继续计算：

- 今日新增玩家
- 今日付费玩家
- 今日付费金额
- 本月新增玩家
- 本月付费玩家
- 本月付费金额
- 历史累计玩家
- 历史累计付费玩家
- 历史累计付费金额

建议结构：

```sql
WITH scoped_roles AS (
  SELECT DISTINCT ba.role_id
  FROM binding_applies ba
  WHERE ba.status = 'approved'
    AND ...scope...
),
role_metrics AS (
  -- 隐含前提：roles.role_id 具有 UNIQUE 约束（schema 中为 @unique），
  -- 因此 INNER JOIN scoped_roles 不会产生重复行，COUNT(*) 即为精确计数。
  SELECT
    COUNT(*) FILTER (
      WHERE r.register_time >= :todayStart
        AND r.register_time < :tomorrow
    )::int AS today_new_players,
    COUNT(*) FILTER (
      WHERE r.register_time >= :monthStart
        AND r.register_time < :nextMonth
    )::int AS month_new_players,
    COUNT(*)::int AS total_players
  FROM roles r
  INNER JOIN scoped_roles sr ON sr.role_id = r.role_id
),
order_metrics AS (
  SELECT
    COUNT(DISTINCT o.role_id) FILTER (
      WHERE o.pay_time >= :todayStart
        AND o.pay_time < :tomorrow
    )::int AS today_paid_players,
    COALESCE(SUM(o.pay_amount_usd) FILTER (
      WHERE o.pay_time >= :todayStart
        AND o.pay_time < :tomorrow
    ), 0)::double precision AS today_paid_amount,

    COUNT(DISTINCT o.role_id) FILTER (
      WHERE o.pay_time >= :monthStart
        AND o.pay_time < :nextMonth
    )::int AS month_paid_players,
    COALESCE(SUM(o.pay_amount_usd) FILTER (
      WHERE o.pay_time >= :monthStart
        AND o.pay_time < :nextMonth
    ), 0)::double precision AS month_paid_amount,

    COUNT(DISTINCT o.role_id)::int AS total_paid_players,
    COALESCE(SUM(o.pay_amount_usd), 0)::double precision AS total_paid_amount
  FROM orders o
  INNER JOIN scoped_roles sr ON sr.role_id = o.role_id
  WHERE o.is_sandbox = false
)
SELECT
  rm.today_new_players,
  rm.month_new_players,
  rm.total_players,
  om.today_paid_players,
  om.today_paid_amount,
  om.month_paid_players,
  om.month_paid_amount,
  om.total_paid_players,
  om.total_paid_amount
FROM role_metrics rm
CROSS JOIN order_metrics om;
```

**与原始版本的关键差异说明**：

1. **时间边界改为左闭右开**：`>= :todayStart AND < :tomorrow`（而非 `<= :todayEnd`），避免 timestamp 精度问题导致边界数据遗漏或重复
2. **`is_sandbox = false` 上提到 `FROM orders o ... WHERE` 基础条件**：原版放在每个 `FILTER` 内重复声明，现改为在 `order_metrics` 的基表 WHERE 中一次过滤，减少 JOIN 后的扫描量，执行计划更优
3. **新增隐含前提注释**：明确标注 `roles.role_id` 的唯一约束是 COUNT 正确性的前提

### 重要实现细节

1. 使用 `DISTINCT ba.role_id`
   - 防止历史脏数据、重复申请记录导致重复统计
   - 同一 roleId 可能有 rejected + approved 两条记录，DISTINCT 确保只计一次

2. 使用 SQL 内 `COALESCE(..., 0)`
   - 确保零数据时直接返回 0
   - 避免后端出现 `null`

3. 使用显式类型转换：
   - `COUNT(...)::int`
   - `SUM(...)::double precision`
   - 避免 Prisma 对 PostgreSQL `numeric` / `bigint` 的返回类型在 Node 侧出现字符串或 `Decimal` 处理混乱

4. 不拼接字符串 SQL
   - 必须用 `Prisma.sql` 参数化，避免 SQL 注入

#### 3.3.3 Prisma.sql 动态条件拼接的具体实现模式

`Prisma.sql` 的 tagged template 不支持像拼接字符串那样随意组合 SQL 片段。要实现"不同角色生成不同 WHERE 条件"，需要使用 `Prisma.sql` 的可组合特性——**将多个 `Prisma.sql` 片段嵌入到外层 `Prisma.sql` 中**。

具体实现：

```ts
import { Prisma } from '@prisma/client';
import type { CurrentUser } from '../../common/interfaces/current-user.interface';

interface DashboardMetricsRow {
  today_new_players: number;
  month_new_players: number;
  total_players: number;
  today_paid_players: number;
  today_paid_amount: number;
  month_paid_players: number;
  month_paid_amount: number;
  total_paid_players: number;
  total_paid_amount: number;
}

private buildScopeCondition(user: CurrentUser): Prisma.Sql {
  if (user.level === 0) {
    // admin: 无额外过滤
    return Prisma.sql`TRUE`;
  }

  if (user.level === 1) {
    // manager: 本组数据（新旧兼容）
    return Prisma.sql`(
      ba.platform = ${user.cpsGroupCode}
      OR ba.team_leader = ${user.username}
    )`;
  }

  // operator: 仅本人
  return Prisma.sql`ba.applicant = ${user.username}`;
}

async getStatistics(currentUser: CurrentUser) {
  const todayStart = dayjs().startOf('day').toDate();
  const tomorrow = dayjs().add(1, 'day').startOf('day').toDate();
  const monthStart = dayjs().startOf('month').toDate();
  const nextMonth = dayjs().add(1, 'month').startOf('month').toDate();

  const scopeCondition = this.buildScopeCondition(currentUser);

  const rows = await this.prisma.$queryRaw<DashboardMetricsRow[]>(Prisma.sql`
    WITH scoped_roles AS (
      SELECT DISTINCT ba.role_id
      FROM binding_applies ba
      WHERE ba.status = 'approved'
        AND ${scopeCondition}
    ),
    role_metrics AS (
      SELECT
        COUNT(*) FILTER (WHERE r.register_time >= ${todayStart} AND r.register_time < ${tomorrow})::int AS today_new_players,
        COUNT(*) FILTER (WHERE r.register_time >= ${monthStart} AND r.register_time < ${nextMonth})::int AS month_new_players,
        COUNT(*)::int AS total_players
      FROM roles r
      INNER JOIN scoped_roles sr ON sr.role_id = r.role_id
    ),
    order_metrics AS (
      SELECT
        COUNT(DISTINCT o.role_id) FILTER (WHERE o.pay_time >= ${todayStart} AND o.pay_time < ${tomorrow})::int AS today_paid_players,
        COALESCE(SUM(o.pay_amount_usd) FILTER (WHERE o.pay_time >= ${todayStart} AND o.pay_time < ${tomorrow}), 0)::double precision AS today_paid_amount,
        COUNT(DISTINCT o.role_id) FILTER (WHERE o.pay_time >= ${monthStart} AND o.pay_time < ${nextMonth})::int AS month_paid_players,
        COALESCE(SUM(o.pay_amount_usd) FILTER (WHERE o.pay_time >= ${monthStart} AND o.pay_time < ${nextMonth}), 0)::double precision AS month_paid_amount,
        COUNT(DISTINCT o.role_id)::int AS total_paid_players,
        COALESCE(SUM(o.pay_amount_usd), 0)::double precision AS total_paid_amount
      FROM orders o
      INNER JOIN scoped_roles sr ON sr.role_id = o.role_id
      WHERE o.is_sandbox = false
    )
    SELECT rm.*, om.*
    FROM role_metrics rm
    CROSS JOIN order_metrics om
  `);

  const row = rows[0];

  return {
    today: {
      newPlayers: row?.today_new_players ?? 0,
      activePlayers: (row?.today_new_players || row?.today_paid_players) ?? 0,
      paidPlayers: row?.today_paid_players ?? 0,
      paidAmount: row?.today_paid_amount ?? 0,
    },
    monthly: {
      newPlayers: row?.month_new_players ?? 0,
      activePlayers: (row?.month_new_players || row?.month_paid_players) ?? 0,
      paidPlayers: row?.month_paid_players ?? 0,
      paidAmount: row?.month_paid_amount ?? 0,
    },
    total: {
      newPlayers: row?.total_players ?? 0,
      activePlayers: (row?.total_players || row?.total_paid_players) ?? 0,
      paidPlayers: row?.total_paid_players ?? 0,
      paidAmount: row?.total_paid_amount ?? 0,
    },
  };
}
```

**关键点**：

- `Prisma.sql` 支持在 tagged template 中嵌入另一个 `Prisma.sql` 片段，嵌入的片段会被正确参数化
- `buildScopeCondition` 返回的是 `Prisma.Sql` 对象而非字符串，避免 SQL 注入
- 日期参数直接传入 `Date` 对象，Prisma 会自动转为参数化 timestamp
- 不使用 `$queryRawUnsafe`

## 4. 必须联动的加固改动（否则统计真相源不稳）

## 4.1 禁止删除 `approved` 申请（强烈建议列为本次必做）

### 涉及文件

- `suzaku-gaming-server/src/modules/audit/audit.service.ts`

### 现有问题

当前 `deleteBindingApply(id)`：

- 只要记录存在就删除
- 不区分 `pending/approved/rejected`

如果 Dashboard 改为按 `approved binding_applies` 统计，则删除 `approved` 记录会直接导致：

- Dashboard 统计减少
- 但 `roles.cpsGroup/cpsVisible` 仍保留为已绑定状态

这会让系统出现"角色看起来已绑定，但 Dashboard 不再统计"的状态分裂。

### 建议改法

本次直接禁止删除 `approved`，只允许删除：

- `pending`
- `rejected`

示例逻辑：

```ts
if (apply.status === 'approved') {
  throw new BadRequestException('已通过的绑定申请不允许删除');
}
```

### 为什么这是本次必做

因为你要求：

> "仅统计 approved 的归因绑定申请"

一旦 `approved` 记录可被任意删除，Dashboard 的统计真相源就不稳定，需求会在运维操作层面被绕开。

### 如果业务以后需要"撤销已通过绑定"

不要复用"删除申请"语义；应单独设计"撤销绑定"流程，并原子处理：

1. 更新 `binding_applies` 的业务状态（例如新增 `revoked`）
2. 回滚 `roles.cpsGroup/cpsVisible/cpsBindTime/cpsBindBy`
3. 必要时同步清空数数平台 CPS 维度

这不属于本次需求，禁止顺手混在本次里做。

## 4.2 不建议把 Dashboard 统计真相源改成 `Role.cpsGroup/cpsVisible`

虽然 `reviewBindingApply(approve)` 会把角色写成：

- `cpsVisible = true`
- `cpsGroup = apply.platform`

但本次需求明确写的是：

> "仅统计审核通过（approved）的归因绑定申请"

因此不建议把 Dashboard 改成"只看 `roles.cpsGroup/cpsVisible`"。

原因：

1. 这会偏离需求原文
2. `Role` 只反映当前绑定状态，不保留申请审批语义
3. 一旦未来有撤销/修正流，`Role` 更适合做结果快照，而不是审批真相源

结论：

- 统计真相源：`binding_applies(status='approved')`
- 角色表：仅作为被关联的角色/订单数据载体

## 5. 数据库索引方案（强烈建议纳入本次）

## 5.1 `binding_applies` 索引补强（优先级高）

### 涉及文件

- `suzaku-gaming-server/prisma/schema.prisma`

### 建议新增索引

在 `BindingApply` 模型上新增：

```prisma
@@index([roleId])
@@index([platform])
@@index([teamLeader])
@@index([status, roleId])
@@index([status, platform])
@@index([status, teamLeader])
```

### 理由

这些索引同时覆盖：

1. 本次 Dashboard 新查询路径
2. 现有 `createBindingApply()` / `checkRoleForBinding()` 中按 `roleId + status in (...)` 的查询

即使 Prisma schema 里暂时不声明联合唯一约束，至少应该先把读取路径索引补齐。

### 可选补强

当前已有 `@@index([applicant])`，若数据量增大且 `operator` 请求多，可升级为：

```prisma
@@index([status, applicant])
```

如果不想删除旧索引，至少新增组合索引。

### 索引创建的生产环境注意事项

在生产数据库上创建索引需要注意：

- PostgreSQL 默认的 `CREATE INDEX` 会**锁表**，阻塞写入
- Prisma migrate 生成的 DDL 默认**不使用** `CONCURRENTLY`
- 如果 `binding_applies` 数据量较大（万级以上），建议：
  1. 在低峰期执行 migration
  2. 或手动用 `CREATE INDEX CONCURRENTLY` 创建索引后，再同步 schema

本次开发环境数据量小，直接 `prisma db push` 即可；生产部署时需评估。

## 5.2 `orders` 与 `roles` 索引暂不强制变更

当前：

- `roles.roleId` 已唯一（@unique）
- `roles.registerTime` 已有索引
- `orders.roleId` 已有索引
- `orders.payTime` 已有索引

在本次 CTE 方案下，数据库已能利用现有索引完成基础过滤。

因此：

- 本次可先不改 `orders/roles` 索引
- 若上线后观测到慢查询，再按实际执行计划补组合索引

## 9. 风险与取舍

## 9.1 本次必须接受的取舍

1. `activePlayers` 仍是旧口径近似值
   - 这是现有技术债
   - 本次只做权限与 approved 隔离，不重定义 KPI

2. 统计基于"当前仍存在的 approved 申请"
   - 这就是为什么必须禁止删除 approved
   - 若未来要支持撤销，应另建流程

## 9.2 若只做"Dashboard 改查询"，不做删除加固，会出现的风险

1. 统计真相源可被管理员直接删除
2. Dashboard 与 `roles.cpsGroup/cpsVisible` 状态分裂
3. 用户看到"已绑定角色"却不在统计中

这不是低概率边角问题，而是结构性问题。

## 9.3 若只用 Prisma 两步法，不用数据库内集合统计

短期能跑，但会有这些风险：

1. `approved roleId` 多时，Node 侧持有大数组
2. 超长 `IN (...)` 查询可读性差、执行计划不稳定
3. 数据库往返次数更多

因此本方案把它定义为"可临时落地，但不作为推荐实现"。

## 9.4 并发一致性分析

审核通过（`reviewBindingApply`）和 Dashboard 查询可能同时发生：

- 审核通过在事务中同时更新 `binding_applies.status` 和 `roles.cpsGroup/cpsVisible`
- Dashboard 的 CTE 查询是独立的只读操作

在 PostgreSQL 默认隔离级别（Read Committed）下，这是安全的：

- Dashboard 要么读到事务提交前的快照（不含刚 approved 的数据），要么读到提交后的完整状态
- 不会出现"申请已 approved 但角色未更新"的中间态

因此本方案无需额外处理并发锁，PostgreSQL 的 MVCC 机制已提供足够保证。

## 9.5 查询性能与缓存策略

Dashboard CTE 查询涉及三张表 JOIN，在以下场景可能产生性能压力：

- 多用户同时访问 Dashboard（每次请求都触发完整查询）
- `binding_applies` / `orders` 数据量持续增长

**建议措施**（可在上线后按需实施）：

1. **短期缓存**：在服务层对结果做 30-60 秒的内存缓存（按 `userId` 区分 key），减少数据库压力
2. **NestJS 内置方案**：可使用 `@nestjs/cache-manager` 或简单的 Map + TTL
3. **前端节流**：Dashboard 组件避免频繁重复请求（当前 `onMounted` 仅调用一次，暂无问题）

本次实施可暂不加缓存，但应在性能监控计划中预留。

## 10. 最终建议（执行口径）

本需求建议按以下口径实施：

1. **Dashboard 的统计真相源改为 `binding_applies(status='approved')`**
2. **权限作用域在 Dashboard 内单独实现，不直接复用 Audit 的列表过滤器**

7. **前端接口与渲染结构保持不变，尽量将改动收敛在后端**

9. **时间边界统一左闭右开，避免 timestamp 精度问题**
